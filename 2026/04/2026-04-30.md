## TypeScript 7 Beta Now Enabled by Default in Visual Studio 2026 18.6 Insiders 3

- **Blog:** Visual Studio
- **URL:** https://devblogs.microsoft.com/visualstudio/typescript-7-beta-now-enabled-by-default-in-visual-studio-2026-18-6-insiders-3

### 要約
Visual Studio 2026 18.6 Insiders 3 で、組み込みの TypeScript SDK が TypeScript 7 Beta に更新されたことを告知しています。Visual Studio の TypeScript と JavaScript サポートに使われるコンパイラと language service が対象です。ビルトイン SDK を使うプロジェクトでは、この更新の影響を受けます。TypeScript プロジェクトだけでなく、npm パッケージを含む ASP.NET Core プロジェクトにも関係します。明示的な TypeScript バージョンを入れていない場合は、Visual Studio がこの新しい SDK を使います。Insiders 向けの preview 変更として、先行して挙動を確認しておく内容です。

### 開発者にとって重要なこと
Visual Studio を使う TypeScript/JavaScript 開発者、ASP.NET Core でフロントエンド資産を扱うチーム、拡張機能作者に関係します。組み込み SDK の更新は、補完・診断・ビルド結果に影響することがあります。明示的にバージョン固定していない環境では、開発体験がすぐ変わる可能性があります。

### アクションプラン
- Visual Studio で使っている TypeScript バージョンがビルトイン依存か確認する。
- **⚠️** バージョン差分で診断や型エラーが変わる可能性があるため、主要な TS/JS プロジェクトを一度確認する。
- 必要ならプロジェクト側で TypeScript のバージョンを明示的に固定する。

## Developing a cross-process reader/writer lock with limited readers, part 3: Fairness

- **Blog:** Oldnewthing
- **URL:** https://devblogs.microsoft.com/oldnewthing/20260430-00/?p=112288

### 要約
クロスプロセスの reader/writer lock を、読者数に上限を設けた状態で構築する連載の第 3 回です。今回は fairness に焦点を当て、排他取得が共有取得に押し負けて starvation する問題を説明しています。共有側が後から来ても先に進めてしまうと、排他側がいつまでも進めない状況が起きます。セマフォの token を 1 つずつ取る実装が、スループット低下と不公平を招くという分析です。低レベル同期プリミティブの設計上の落とし穴を扱う記事です。

### 開発者にとって重要なこと
システムプログラミング、同期制御、マルチプロセス設計に関わる開発者に有用です。似た構造の lock を実装する場合、性能だけでなく starvation と公平性を見落としやすいことが分かります。排他処理が遅い原因の切り分けにも役立ちます。

### アクションプラン
- 類似の lock 実装があるなら、共有取得が排他取得を飢餓化させないか確認する。
- トークン取得方式を見直し、公平性とスループットの両方を評価する。
- 同期プリミティブを設計するチームは、再現しやすい負荷条件で検証する。

### リファレンス
- [Developing a cross-process reader/writer lock with limited readers, part 3: Fairness](https://devblogs.microsoft.com/oldnewthing/20260430-00/?p=112288)

## Propagating SharePoint Document Permissions to AI Search and RAG Pipelines

- **Blog:** Ise
- **URL:** https://devblogs.microsoft.com/ise/sharepoint-doc-level-access

### 要約
SharePoint のコンテンツを Azure AI Search や RAG pipeline、Copilot extension に流すときに、文書レベルのアクセス制御を保つ方法を紹介しています。最小権限のために Microsoft Graph の Sites.Selected を使い、取り込み時に権限情報を検索インデックスへ反映する構成です。クエリ時に authorization を強制することで、機密性の高い文書を安全に扱う設計を示しています。SharePoint の権限を下流システムへ伝播させる security-first なアーキテクチャです。AI 検索とアクセス制御を両立したい現場向けの内容です。

### 開発者にとって重要なこと
SharePoint を検索や RAG に取り込むチーム、Copilot extension を作る開発者、機密情報を扱う組織に直接関係します。権限伝播を省くと、情報漏えいリスクが高まります。検索インデックス設計と認可設計を別物として扱わず、同時に実装する必要があります。

### アクションプラン
- **⚠️** SharePoint 由来の検索/RAG では、文書レベルの権限伝播を必須要件として確認する。
- 取り込み時の権限マテリアライズと、クエリ時の authorization を両方実装する。
- Sites.Selected のような最小権限設定を採用し、アクセス範囲を定期的に見直す。

### リファレンス
- [Propagating SharePoint Document Permissions to AI Search and RAG Pipelines](https://devblogs.microsoft.com/ise/sharepoint-doc-level-access)
