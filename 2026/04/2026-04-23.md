## CodeAct in Agent Framework: Faster Agents with Fewer Model Turns

- **Blog:** Agent Framework
- **URL:** https://devblogs.microsoft.com/agent-framework/codeact-with-hyperlight

### 要約
Agent Framework に CodeAct サポートが追加され、AI エージェントの複数回の tool call を 1 回の実行コードにまとめられるようになったことを紹介しています。これにより、モデルの往復回数を減らし、遅延とトークン使用量を大きく抑えられると説明しています。実行環境には agent-framework-hyperlight の alpha パッケージを使い、各呼び出しを Hyperlight の隔離された micro-VM で実行します。記事では、従来の逐次 tool-calling と CodeAct の違いを同じ課題で比較し、どのようなワークロードで効果が出るかを示しています。安全性や隔離性を保ちながら、データ取得や軽い計算を伴うエージェントの効率を上げたい開発者向けの内容です。

### 開発者にとって重要なこと
Agent Framework で AI エージェントを構築しているチーム、特に多数の小さな tool call を連ねるワークフローを持つ開発者に関係します。応答速度、トークンコスト、監査しやすさを改善したいケースで有効ですが、alpha 品質であり、Linux/Windows などの対応条件も確認が必要です。既存の approval モデルや sandbox 設計にも影響するため、導入前に運用方針を整理する価値があります。

### アクションプラン
- いまのエージェントが連続した細かな tool call に依存していないか確認する。
- CodeAct が有効そうな処理を洗い出し、alpha パッケージで小さく検証する。
- ⚠️ approval 要件や sandbox の制約があるため、既存の安全ポリシーと合うかを先に確認する。
- Linux/Windows での実行条件と、既存ツールの安全性・決定性を確認してから本番導入を判断する。

## Another crash caused by uninstaller code injection into Explorer

- **Blog:** Oldnewthing
- **URL:** https://devblogs.microsoft.com/oldnewthing/20260423-00/?p=112261

### 要約
32-bit Explorer 上で発生したクラッシュの調査から、アンインストーラーに注入されたコードが原因だったという話です。問題の本質は、関数呼び出しで正しい calling convention を指定していなかったことにあり、スタックが少しずつ壊れて最終的にコード領域まで侵食していました。記事では、__stdcall と __cdecl の違いがクラッシュに直結した経緯を解説しています。結果として、実行中コードがスタックで上書きされ、無効命令で落ちるという形になりました。ネイティブコード、特に Windows へのコード注入や古い互換実行環境に関わる開発者にとって注意喚起になる内容です。

### 開発者にとって重要なこと
Windows のネイティブ開発、アンインストーラー、シェル拡張、注入コードを扱う開発者に関係します。calling convention の不一致はクラッシュだけでなく、診断を難しくする再現性の低い不具合にもつながります。32-bit/64-bit の混在環境で動くコードは、ABI の取り違えが特に危険です。

### アクションプラン
- ネイティブ関数の宣言で calling convention が実装と一致しているか見直す。
- 32-bit/64-bit 混在環境で、アンインストーラーや注入コードの動作を再確認する。
- ループ内で再試行する処理がスタックやメモリを壊していないか、エラーハンドリングを点検する。
- ⚠️ Explorer への注入や互換プロセスを扱うコードは、クラッシュ解析ログを残して回帰検知できるようにする。
